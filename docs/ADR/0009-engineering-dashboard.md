# ADR 0009 — Engineering Dashboard

**Date:** 2026-07-20 (revised same day — see Revision Note)
**Status:** Proposed
**Deciders:** AI Engineering Team

> **Revision note:** Section registration is now keyed by artifact-type
> category, using the exact same `CategoryRegistry[T]`
> (`src/quality/registry.py`, ADR-0004 Decision 6b) that backs the metric
> registry — not a second, independently-written registry class, and not
> the single flat `DATASET_SECTIONS` list with an informally-sketched
> "parallel `MODEL_SECTIONS` list" for later that the first draft of this
> ADR had. A seventh section — Lineage (ADR-0011) — is added to the
> `"dataset"` category now, ahead of Phase 4, as the first real exercise
> of this mechanism.

---

## Context

By the time ADR-0004 through ADR-0008 are implemented, a dataset has: a
quality score (ADR-0005), a version record (ADR-0006), a regression report
against its predecessor (ADR-0007), and coverage recommendations
(ADR-0008) — four separate JSON files a reviewer would otherwise have to
open one at a time. The brief's Definition of Done requires answering
"is this dataset trustworthy," "what changed," "is quality improving,"
and "is training allowed" **without manually inspecting JSON files**. This
ADR defines `src/quality/dashboard.py`, the single human-facing view over
everything Phase 3.5 produces.

---

## Decisions

### 1. The dashboard is a single self-contained static HTML file, generated on demand — not a running server

**Decision:** `dashboard.py::generate_dashboard(dataset_dir, cfg) ->
Path` writes one `.html` file (default
`outputs/dashboard/<dataset_id>_dashboard.html`, path configurable) with
all CSS inlined and zero external requests (no CDN scripts, no fonts, no
JS framework). Charts (Decision 4) are rendered as inline SVG generated by
Python, not by a client-side charting library.

**Rationale:**
- Consistent with ADR-0004 Decision 3: no new service, no database, no
  daemon. A static file opens in any browser on macOS/Linux/Windows with
  zero setup, satisfies the Engineering Principles' portability
  requirement (including "future cloud execution," where an offline
  artifact is trivially archivable/shippable as a CI build artifact),
  and needs no new runtime dependency — the entire repository's
  dependency set already includes everything needed to write HTML/SVG
  strings (the standard library is sufficient; no `jinja2`, no
  `plotly`, no npm toolchain).
- A self-contained file (no external CDN) also means the dashboard still
  renders correctly on an air-gapped CI runner or a machine with no
  internet access — consistent with this repository's CARLA-optional,
  network-optional design elsewhere (e.g. `diagnose.py` never assumes
  internet access).

**Consequences:** `make dashboard` is a pure batch step — "regenerate the
file," not "refresh a live view." A reviewer who wants a fresher dashboard
re-runs the command; there is no cache-invalidation problem to design
around because there is no cache.

### 2. The dashboard is built from an ordered list of pluggable sections, not a single fixed template

**Decision:**

```python
@dataclass
class DashboardSection:
    title: str
    order: int
    render: Callable[[DashboardContext], str]   # returns an HTML fragment

from src.quality.registry import CategoryRegistry

SECTION_REGISTRY: CategoryRegistry[DashboardSection] = CategoryRegistry()
# the identical class metrics.py::METRIC_REGISTRY uses — see ADR-0004 Decision 6b

# registered by dashboard.py itself at import time, category "dataset":
#   Header, Quality, Coverage, Validation Gate, Recent Changes, Quality Trend, Lineage (ADR-0011)
```

`generate_dashboard(artifact_dir, cfg)` resolves the artifact's
`artifact_type` (from its `version.json`, or `"dataset"` by default when
none exists yet), builds one `DashboardContext`, and renders every section
registered under that category, in `order`, concatenating the HTML
fragments into one page.

**Rationale:**
- This is the deliverable's explicit requirement: "Design so future phases
  can add Model Metrics, Inference Metrics, Simulation Metrics, Deployment
  Metrics without architectural changes." A fixed template with seven
  hardcoded sections would require editing `dashboard.py`'s core rendering
  function every time a later phase adds a section. Keying the section
  list by category (rather than one flat list plus an ad hoc "parallel
  list" per artifact type) is the same fix ADR-0004 Decision 6b already
  made for metrics, applied here for the identical reason: one registry
  object, queried by category, instead of a new global constant per
  artifact type. Phase 4 calls
  `SECTION_REGISTRY.register("model", ModelMetricsSection)`; nothing about
  `generate_dashboard()`'s control flow changes.
- Keeping `render()` as "context in, HTML fragment out" means a section
  can be unit-tested in isolation (assert a fragment contains an expected
  string) without rendering the whole page — matching the repository's
  existing bias toward small, independently testable functions.

**Consequences:** Today's seven sections (Dataset identity, Quality,
Coverage, Validation, Version/Recent Changes, Quality Trend, Lineage) are
the initial contents of category `"dataset"`; nothing about the
registry's own mechanism changes when an eighth section — or a whole new
category — is added later.

### 3. The dashboard never computes anything — it only formats what `scoring.py` / `versioning.py` / `regression.py` / `coverage.py` / `gates.py` / `lineage.py` already returned

**Decision:** Every section's `render()` function receives already-computed
dataclasses (`QualityScore`, `VersionRecord`, `RegressionReport`,
`list[CoverageRecommendation]`, `GateReport`, `LineageGraph` traversal
results) and formats them as HTML. It never re-derives a score, threshold
check, comparison, or lineage edge.

**Rationale:** Direct application of ADR-0004 Decision 2's
"composition-only" rule to the dashboard specifically — this is the
deliverable most tempted to accumulate its own logic (it is, after all,
the most visible surface), so the boundary is stated explicitly here as
well as in ADR-0004. A dashboard bug is then always either a rendering bug
(isolated to `dashboard.py`) or a computation bug (isolated to exactly one
of the five other modules) — never both at once.

**Consequences:** If `dashboard.py`'s quality-trend section and
`dataset_quality.py`'s CLI output ever disagreed on a dataset's grade,
that would be a genuine bug (both read the same `quality_score.json`) —
there is no code path where they could legitimately diverge.

### 4. Quality trend is rendered as a plain inline SVG line chart built from historical `version.json` / `quality_score.json` files, not a third-party charting library

**Decision:** The "Quality Trend" section scans `datasets_dir` for every
dataset directory containing both `version.json` and `quality_score.json`,
orders them by `created_at`, and renders their `overall_score` values as a
minimal hand-generated `<svg>` polyline (a few dozen lines of coordinate
math) plus a plain HTML table of the same values as a text fallback.

**Rationale:**
- Reuses exactly the artifacts ADR-0005/ADR-0006 already produce — no new
  "trend history" file is maintained separately (which would be another
  place for staleness to creep in, the same failure mode ADR-0006
  Decision 5 rejected for changelogs).
- A hand-rolled SVG polyline needs no dependency and is trivially testable
  (assert the right number of `<circle>`/`<path>` points appear for N
  historical datasets) — a real charting library would be the single
  heaviest dependency added anywhere in this phase for a repository that
  has otherwise stayed deliberately dependency-light since Phase 3a (ADR-
  0003 Decision 2 made the identical trade-off for duplicate detection).
- The plain-table fallback ensures the same information is available even
  to a text-only viewer or a terminal-based Markdown renderer, and gives
  the golden-output tests something simple to assert against alongside the
  SVG.

**Consequences:** The trend chart is deliberately simple (a single-series
line, no zoom/hover/interactivity) — sufficient to answer "is quality
improving release over release" at a glance, which is the literal
requirement, without building a general-purpose charting component this
project does not otherwise need.

### 5. "Future Training Readiness" renders `gates.py`'s `GateReport` verbatim, including failing checks

**Decision:** The Validation section always shows every configured gate
check (ADR-0004 extension point 3) with its pass/fail status and reason,
even when the overall verdict is "pass" — not just a single green/red
summary badge.

**Rationale:** A reviewer acting on "training is blocked" needs to know
*which* check failed and why, immediately, without cross-referencing
`quality_report.json` — this is the dashboard's most direct answer to the
brief's "Why or why not?" requirement under Definition of Done.

**Consequences:** Adding a new gate check (e.g. a future Phase 4
model-readiness check per ADR-0010) automatically appears in this section
with no `dashboard.py` change, because it renders whatever list
`gates.py` returns rather than a hardcoded set of named checks.

### 6. A Lineage section renders ancestor/descendant traversal from `lineage.py`, and is the first section registered by a module other than `dashboard.py` itself

**Decision:** `lineage.py` (ADR-0011) registers its own `LineageSection`
into `SECTION_REGISTRY` under category `"dataset"`, rather than
`dashboard.py` importing lineage-rendering logic directly.

**Rationale:** Every other section today happens to be written inline in
`dashboard.py` because every other computing module (`scoring.py`,
`versioning.py`, `regression.py`, `coverage.py`, `gates.py`) predates the
dashboard and `dashboard.py` was free to import all of them directly at
once. `lineage.py` is the first case where a *separate* module registers
its own section from its own file — proving the registry mechanism
(Decision 2) works as an actual cross-module extension point, not just a
list `dashboard.py` happens to hardcode in one place. This is exactly the
shape Phase 4/5/6's `ModelMetricsSection` / `SimulationMetricsSection` /
`DeploymentMetricsSection` will follow.

**Consequences:** `dashboard.py` must import `src.quality.lineage` for its
`register()` side effect to run (the same import-time registration
pattern `dataset_metrics.py` already uses for `METRIC_REGISTRY` —
ADR-0005 Decision 1) — documented explicitly in `docs/QUALITY_SYSTEM.md`
so a future module author doesn't miss the import and wonder why their
registered section never appears.
